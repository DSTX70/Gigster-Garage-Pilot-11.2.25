# Gigster Garage Pod Catalog + Shared System Primitives

This canvas captures our full working spec for how Dream Team Pods/Members/SMEs/Councils operate in **Gigster Garage** (and how the same system standardizes across other repos such as **Agency Hub**).

---

## Operating Model

### What a Pod is

A **Pod** is a governed workflow (a production pipeline), not “many bots.”

Each Pod consists of:

* **Roles** (Members/SMEs/Reviewers) with clear **input contracts** and **output contracts**
* **Routing rules** (how requests map to Pods)
* **Autonomy + gates (L0–L3)** (what can run without explicit approval)
* **Adjudication rules** (how conflicts are resolved)
* **Artifact requirements** (packs + receipts)

### Members vs SMEs vs Councils

* **Members (Core)**: always-on roles needed to deliver the product’s promise.
* **SMEs (Optional/Targeted)**: invoked only when triggers indicate their lane is needed.
* **Councils**: invoked when **risk/conflict/irreversibility** requires formal review. Councils adjudicate **decisions**, not personalities. They *can* adjudicate core outputs when thresholds are hit.

### “Tracks → Packs” translation

Parallax Translate surfaces **tracks** (Literal/Singable/Poetic). Gigster Garage surfaces **packs** (Profile, Proposals, Rate Card, Safety Plan, etc.).

# Shared System Primitives (used by every Pod)

## A) Learning Artifacts (explicit, portable)

### 1) UserProfilePack

Stable, user-approved preferences and constraints.

**Suggested fields**

* `userId`
* `preferredName`
* `timezone`
* `preferredTone` (e.g., direct/warm/formal)
* `verbosity` (short/medium/long)
* `goals[]`
* `constraints[]` (e.g., “avoid optimization language”, “don’t be salesy”, “no emojis”)
* `riskTolerance` (low/medium/high)
* `defaultDeliverableFormats[]` (e.g., packs/checklists/scripts)

### 2) PreferenceSignals

Implicit learning derived from selections/edits/rejections.

**Suggested fields**

* `signalId`
* `podId`
* `chosenVariantId`
* `editDeltasSummary`
* `rejectionReasons[]` (enum list)
* `timeToApprovalSec`
* `shipActionTaken` (boolean)
* `lastNPatterns` (derived preferences)

### 3) RunReceipt

A traceable receipt of what happened (supports QA, consistency, and evidence-grade posture).

**Suggested fields**

* `runId`
* `podId`
* `workType`
* `inputsSummary`
* `contextSourcesUsed[]` (explicit prompt, page fields, user profile pack, etc.)
* `rolesInvoked[]` (core + SMEs)
* `riskScore` + breakdown
* `councilInvoked` (yes/no)
* `outputsProduced[]` (pack names + ids)
* `decisionLogId`
* `userApproval` (approved/rejected + notes)

### Constraint-loading rule (global)

Pods load constraints in this priority order:

1. **Page/form fields** (highest trust)
2. **Explicit user prompt**
3. **Project canon/policies** (brand lock, repo rules)
4. **UserProfilePack**
5. **PreferenceSignals** (style/format tuning only; never override explicit constraints)

---

## B) FactCheck Lane (VERIFY) — standardized

You can implement as either:

* `factCheckSME` role invocation, **or**
* a dedicated **VERIFY step** embedded in relevant Pods.

### Required outputs

#### 1) ClaimsList

Extract factual claims that should be verified.

**Fields**

* `claimId`
* `claimText`
* `claimType` (law/policy, pricing, stats, “latest/current”, medical-ish, platform rules)
* `timeSensitivity` (low/med/high)
* `verifyRequired` (bool)

#### 2) VerifiedClaims

Verification outcomes.

**Fields**

* `claimId`
* `status` (verified / contradicted / uncertain)
* `supportSummary`
* `sourceRefs[]` (citations or internal refs)
* `confidence` (low/med/high)
* `notes`

#### 3) UnverifiableFlags

Claims that must be reframed as opinion/estimate or explicitly marked unknown.

**Fields**

* `flagId`
* `claimText`
* `reason` (no sources, time-sensitive, ambiguous, user-specific unknown)
* `rewriteGuidance`

### When VERIFY runs (global)

Run VERIFY when any are true:

* outputs include **laws/regulations/taxes/contracts/medical claims**
* outputs include **prices/platform policies/“latest/current” facts**
* outputs include **stats/benchmarks** or “X is best”
* **risk score** hits Council threshold (below)

---

## C) Decision Logging

### DecisionLogPack (required when adjudication occurs)

**Fields**

* `decisionId`
* `conflictSummary`
* `options[]`
* `precedenceApplied[]`
* `finalChoice`
* `notes`

# Cross-Repo Routing + Escalation Standard

## 1) WorkType classification

Every AI entry point assigns `workType`.

**Portable WorkTypes**

* `PROFILE_OPTIMIZATION`
* `PROPOSAL_WRITING`
* `PRICING_SCOPING`
* `SCAM_SAFETY_CHECK`
* `CONTRACT_WATCHOUTS`
* `FINANCE_TAX_PLANNING`
* `PIVOT_PLAN`
* `CONTENT_CREATE`
* `CONTENT_WRITE`
* `PROMOTION_PLAN`
* `ANALYTICS_TRACKING`

**Routing hierarchy (global)**

1. If user requests a **ship action** (publish/send/submit/apply/commit), route to the Pod that owns the ship action and raise risk.
2. Use page context (Create/Write/Promote/Track, GigsterCoach, etc.) to narrow candidates.
3. Classify by keywords + required outputs.

---

## 2) SME enrollment triggers (trigger phrases → SME)

### sentinelSecuritySME (scams/PII/unsafe behavior)

**Phrase triggers**: SSN, social security, driver’s license, passport, pay to apply, wire transfer, gift cards, crypto, deposit first, share login, password, 2FA, TeamViewer, AnyDesk, remote access.
**Structural triggers**: uploading IDs/tax docs/bank info; any payment required before job.

### praetorPrivacySME (privacy/data handling)

**Phrase triggers**: store customer data, PII, GDPR, HIPAA, consent, data retention.
**Structural**: user wants to collect/store client data; analytics/tracking includes personal data.

### aegisLegalIPSME (contracts/IP watchouts; non-lawyer guidance)

**Phrase triggers**: contract, NDA, non-compete, work for hire, ownership, indemnify, legal notice.
**Structural**: any contract text pasted/uploaded.

### ledgerMoneySME (pricing/margins)

**Phrase triggers**: how much should I charge, rate, pricing, quote, scope creep.
**Structural**: job requires estimate/tiering.

### tallyTaxSME (tax education)

**Phrase triggers**: 1099, estimated taxes, quarterly, deductions.
**Structural**: income variability planning.

### accessibilitySME

**Phrase triggers**: reading level, accessible, screen reader, alt text.
**Structural**: public distribution content.

### localizationSME / terminologySME

**Phrase triggers**: translate, bilingual, Spanish version, terminology.
**Structural**: multilingual profile/services.

### codaOSSComplianceSME

**Phrase triggers**: open source, GPL, MIT license, dependencies.
**Structural**: deliverable includes software distribution.

### factCheckSME (VERIFY)

Triggered by VERIFY conditions.

---

## 3) Risk scoring → Council threshold

### Risk dimensions (0–3 each)

* **LEGAL**
* **MONEY**
* **SAFETY/SCAM**
* **PRIVACY**
* **IRREVERSIBILITY** (publish/send/submit/apply/commit)
* **UNCERTAINTY** (missing inputs/underdetermined)
* **RECENCY** (time-sensitive “latest/current” claims)

**Total Risk Score = sum (0–21).**

### Council thresholds

* **0–5:** Core Members only; OS adjudicates.
* **6–10:** SMEs as needed; VERIFY recommended if factual claims present.
* **11–14:** Council required **or** dual-SME signoff (e.g., Sentinel + Aegis).
* **15+:** Council required + VERIFY required + explicit ship gate.

---

## 4) Precedence rules for adjudication

Adjudication resolves **decisions**, not “who wins.”

**Precedence (highest → lowest):**

1. Hard safety/privacy policies (cannot-do)
2. Repo/brand canon (locked systems/templates/disclaimers)
3. User explicit constraints
4. Domain correctness (SME constraints)
5. Core Member quality preferences (style/persuasion/creativity)
6. Optional enhancements

# Gigster Garage Pod Catalog (one page per Pod)

## POD 01 — Worker Success Pod

**PodId:** `GIG_WORKER_SUCCESS_POD`

**Purpose:** Increase replies/bookings/earnings via profiles, proposals, outreach, and pricing alignment.

### Triggers

* Page context: GigsterCoach, Profile, Apply, Messages, Rates
* Phrases: “no replies”, “fix my profile”, “better proposals”, “how do I price this”, “scope this job”
* WorkTypes: `PROFILE_OPTIMIZATION`, `PROPOSAL_WRITING`, `PRICING_SCOPING`

### Steps

1. INTAKE
2. LOAD CONSTRAINTS (UserProfilePack + PreferenceSignals)
3. DRAFT (Core)
4. SME PASSES (conditional)
5. VERIFY (if rules hit)
6. ADJUDICATE (DecisionLogPack)
7. FINALIZE PACKS
8. SHIP GATE (approval for send/apply/submit)
9. WRITE RECEIPT (RunReceipt + PreferenceSignals update)

### Core Members

* OS (orchestrator/adjudicator)
* Prism (positioning)
* Echo (copy + sequences)
* Ledger (pricing sanity)
* Navi (workflow/checklists)

### Optional SMEs

* sentinelSecuritySME, praetorPrivacySME
* factCheckSME (VERIFY)
* accessibilitySME
* localizationSME / terminologySME

### Outputs (Pack schema names)

* `ProfileDeltaPack`
* `ProposalPack`
* `OutreachSequencePack`
* `RateCardPack`
* `ScopeBoundaryPack`
* `ApplyRoutineChecklistPack`
* `DecisionLogPack`
* `RunReceipt`
